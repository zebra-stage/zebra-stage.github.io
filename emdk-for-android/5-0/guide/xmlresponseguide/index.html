EMDK XML programmer's guide

EMDK For Android 5.0

Introduction

This document provides the overview on Profile XML, applying profiles, interpreting response returned the Profile Manager Methods and the response XML schema for the developer to understand and configure the device based their application specific requirements.

Profile XML

When the developer creates the profile based on their configuration requirement in the Profile Manager wizard in IDE, selects and configures features, the settings will be saved under the application’s assets folder as “EMDKConfig.xml”.

A sample Profile XML (e.g., EMDKConfig.xml), show below, was generated by the Profile Manager Wizard using Clock and AppMgr features:

    
    <?xml version="1.0" encoding="UTF-8"?>
    <!--This is an auto generated document. Changes to this document may cause incorrect behavior.-->
    <wap-provisioningdoc>
    <characteristic type="ProfileInfo">
        <parm name="created_wizard_version" value="5.0.1"/>
    </characteristic>

    <characteristic type="Profile">
        <parm name="ProfileName" value="MyProfile1"/>
        <parm name="ModifiedDate" value="2016-08-17 09:56:17"/>
        <parm name="TargetSystemVersion" value="4.2"/>

        <characteristic type="AppMgr" version="4.2">
        <parm name="emdk_name" value="DemoInstallation"/>
        <parm name="Action" value="Install"/>
        <parm name="APK" value="/sdcard/MyDemo.apk"/>
        </characteristic>

        <characteristic type="Clock" version="4.2">
        <parm name="emdk_name" value="MyClock"/>
        <parm name="AutoTime" value="false"/>
        <parm name="TimeZone" value="GMT-5"/>
        <parm name="Date" value="2016-08-31"/>
        <parm name="Time" value="23:55:55"/>
        </characteristic>

    </characteristic>
    </wap-provisioningdoc>

  • An EMDKConfig.xml begins with node <wap-provisioningdoc> and ends with </wap-provisioningdoc>

  • After the opening <wap-provisioningdoc> tag will be a characteristic node that provides information on the Profile Manager Wizard version used to create the profile. This characteristic node will be excluded from the XML document when the profile is applied do a device.

    
<characteristic type="ProfileInfo">
    <parm name="created_wizard_version" value="5.0.1"/>
</characteristic>

  • Next would be a characteristic node with the type of Profile( shown below ). An EMDKConfig.xml can have one or more Profile nodes.

    
<characteristic type="Profile">
    <!-- Profile contents -->
</characteristic>.

  • Each Profile node will have parm nodes for profile name, profile modified date and target MX version used to create profile.

  • The profile name will be used to inform the ProfileManager methods to apply specific profile on the device. The modified date and target MX version are used for profile maintenance such as editing, renaming and modifying profiles in the profile manager wizard. This will not be set on the device.

  • Each profile will have one or more characteristic feature nodes with parm settings specific to that feature. An example is show below. 

    
<characteristic type="AppMgr" version="4.2">
    <parm name="emdk_name" value="DemoInstallation"/>
    <parm name="Action" value="Install"/>
    <parm name="APK" value="/sdcard/MyDemo.apk"/>
</characteristic>

  • The characteristic node for each feature will have the attributes type and version.

    • The type indicates the feature
    • The version indicates the feature specification version

  • Each feature characteristic node will have one or more parm nodes which define the settings for that feature. The parm node (emdk_name) provides the name assigned to that feature which will be used by the Profile Manager API to identify feature and must be a unique name within that profile.

Applying Profiles

The ProfileManager API exposes methods (processProfile and processProfileAsync) that are used to apply a XML profile located in your Application’s Assets folder and/or optional profile data passed as extraData to these methods. 


EMDKResults results = profileManager.processProfile(String profileName,
               ProfileManager.PROFILE_FLAG profileFlag,
               String[] extraData);

processProfile is blocking call which blocks the main thread. To avoid ANRs while applying profile, a developer should call processProfile on a worker thread or use the non blocking call processProfileAsync which returns the status through the registered callback.

ProcessProfile and processProfileAsync methods return an EMDKResults object that provides the status of the requested profile action.

Interpreting Profile Response

An EMDKResults object provides a status of whether the submitted profile was successfully processed, or if there were errors while processing.

The first step to determine if profile processing was a success or failure would be to inspect the EMDKResults.statusCode field. This field will be of type EMDKResults.STATUS_CODE. An EMDKResults.statusCode equal to FAILURE, indicates an error occurred and that more information on the error can be obtained by inspecting EMDKResults.extendedStatusCode. EMDKResults.statusCode could also be SUCCESS or CHECK_XML. 


EMDKResults results = profileManager.processProfile(String profileName,
            ProfileManager.PROFILE_FLAG profileFlag,
            String[] extraData);

If(results.statusCode == STATUS_CODE.SUCCESS ) {
//Success
} else If(results.statusCode == STATUS_CODE.FAILURE )  { 
//Error occurred, check results. extendedStatusCode to more info on the error
} else If(results.statusCode == STATUS_CODE. CHECK_XML )  { 
//parse the response xml to ensure there are no errors
String responseXML = results.getStatusString();
//…
}

In many cases, the Status Code will be CHECK_XML regardless of whether there was a processing error or not. This status code does not clarify if a processing failure happened, only that the XML response must be checked for errors. The CHECK_XML status code should be used to determine if XML processing is necessary.

XML - Without Errors

In this example, using Profile Manger, a new profile has been created and a Clock feature has been added. The parms for the clock feature ( emdk_name,AutoTime,TimeZone,Date,and Time) have been set with values. Profile Manager will error check and provide feedback as values are entered into each parm field. This ensures that the values entered are properly formatted.

Note: Profile Managers ability to error check parm values is restricted to form fields that have a defined schema such as the Time and Date. Free form fields can not validated.

Parsing Response XML

Parsing the Response XML will require the use of a XML Parser library, in this example we will use org.xmlpull.v1.XmlPullParser. XmlPullParser will allow us to quickly iterate over each START_TAG in the XML response looking for Parm and Characteristic errors.

Characteristic-Error - A Characteristic-Error is an element that appears in a Result XML document that is used to indicate a failure when processing a corresponding characteristic in a Request XML document.

Parm-Error - A Parm-Error is an element that appears in a Result XML document that is used to indicate a failure when processing a corresponding parm in a Request XML document.

  1. First import the XML parsing library

    
import org.xmlpull.v1.XmlPullParser;
import org.xmlpull.v1.XmlPullParserException;

  2. Next we will get a string representation of the XML Response, Then instantiate a new XMLPullParser object, and set its input to the the XML Response String object.

    
String xmlResponse = results.getStatusString();
XmlPullParser parser = Xml.newPullParser();
parser.setInput(new StringReader(xmlResponse));

  3. Finally we will use XmlPullParser events to iterate over the XML Response nodes searching for parm and characteristic errors. In this example, I simply log the errors for later review. You will need to add custom error handling to each of the if blocks inside the START_TAG case if logging is not sufficient.

    
try {
    int event;
    try {
        event = parser.getEventType();
        while (event != XmlPullParser.END_DOCUMENT) {
            String name = parser.getName();
            switch (event) {
                case XmlPullParser.START_TAG:


                    if (name.equals("parm-error")) {
                    parmName = parser.getAttributeValue(null, "name");
                    errorDescription = parser.getAttributeValue(null, "desc");
                    Log.d(TAG," (Name: " + parmName + ", Error Description: " + errorDescription + ")");
                    return;
                }
                if (name.equals("characteristic-error")) {
                    errorType = parser.getAttributeValue(null, "type");
                    errorDescription = parser.getAttributeValue(null, "desc");
                    Log.d(TAG," (Type: " + errorType + ", Error Description: " + errorDescription + ")");
                    return;
                }
                break;
            case XmlPullParser.END_TAG:

                break;
        }
        event = parser.next();
    }

    } catch (Exception e) {
        e.printStackTrace();
    }


} catch (XmlPullParserException e) {
    e.printStackTrace();
}

    

} else {
    //Profile processed sucessfully
}



}
