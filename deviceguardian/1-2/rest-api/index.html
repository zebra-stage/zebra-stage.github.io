REST APIs

Device Guardian + Access Management 1.2

All references to Device Guardian also apply to Device Guardian Access Management within this guide.

Overview

The Device Guardian REST APIs provide a comprehensive set of features for managing resources, including sites, access points, and devices. These APIs enable key operations such as creating, updating, and deleting resources, as well as managing device recommissioning and decommissioning.

Key Features

  • Resource Management - Perform create, update, and delete operations for multiple resources in a single API request, streamlining management tasks.
  • Batch Processing - Typically handle up to 300 resource operations - spanning devices, access points, and sites - in a single API call, ensuring efficient large-scale processing.
  • Secure and Authenticated Access - All API requests require an authentication token to ensure secure communication between your application and the API. The authentication process leverages OAuth for enhanced security, providing controlled and authorized access.

Authentication Mechanisms

The Device Guardian REST APIs provide two distinct authentication mechanisms, tailored to meet specific roles and use cases:

  • Partner Role (3-Legged OAuth) - Designed for third-party partner applications acting on behalf of customers. This method requires explicit customer authorization through Zebra Single Sign-On (SSO) and ensures secure, customer-approved access to resources.
  • Customer Role (2-Legged OAuth)- Geared toward customers managing their own resources directly. This simpler authentication flow eliminates the need for user interaction, relying on pre-configured credentials for server-to-server communication.

Each role is designed to meet the specific needs of its audience, balancing security with ease of use. The access request and authentication processes differ depending on the role, so it is important to follow the appropriate guidelines.

Request Access

Before using the REST APIs, access must be requested. Choose the appropriate process based your use case: Partner or Customer.

Partner Access Request

For Partners to request API access for both themselves and their customers, follow these steps:

  1. Register an Account - Visit the Zebra Developer Portal to register.
  2. Submit the Request:
    • Navigate to the Contact Us page and click Zebra Enterprise Support.
      • Fill out the support form:
        • For the Product Serial Number field, enter a valid device serial number.
        • Enter "Device Guardian API Access" for the Subject.
        • In the Description field, specify:
          • The partner email address from the initial API access setup.
          • The customer's Device Guardian web portal address or GCP project ID
          • State that this is for a partner that already has API access but needs to add customer access.
          • Provide a list of customer email addresses that require access.
  3. Retrieve Credentials - After you receive confirmation that access is granted, log in to the Zebra Developer Portal to retrieve your application credentials:
    • Login to the Zebra Developer Portal.
    • Navigate to Data Services > API List.
    • Click the 3-dot menu next to the approved app and select View Details.
    • Copy the Consumer Key and Consumer Secret.

Requirements

Requirements for API use:

  • Base URL - All API requests must be directed to the designated base URL, which serves as the foundational endpoint for all API requests related to the Device Guardian services. Ensure that API calls are constructed using the correct base URL to retrieve the intended resources and functionalities. It is critical to use the appropriate URL based on the user role to ensure proper Rest API operation:
    • For Customers Only: https://api.zebra.com/device-guardian
    • For Partners Only: https://api.zebra.com/partner-device-guardian
  • API Version - The current version of the API is v1.0.0. Each API response includes this version number in the result, ensuring consistency and compatibility with your application. Versioning is important for maintaining alignment with API updates and changes. Verify the API version to avoid disruptions caused by potential updates.
  • Request Body (bodyData) - Every request must include a JSON object as the body of the request. This object contains the data and parameters required for the API to perform the desired operation, ensuring the request is processed correctly.
  • Access Token - An access token is required for all API requests and must be included in the Authorization header. It is essential for authorizing and securing API calls, ensuring system integrity and restricting access to authenticated users only.

Authenticate

The Authenticate API facilitates secure access to APIs by implementing the OAuth authentication flow. It provides both partners and customers with a trusted method to authenticate and interact with APIs while protecting sensitive credentials. Authentication through OAuth is mandatory for accessing any REST API, ensuring that all interactions between your application and the API are secure and protected.

Credentials Required:

  • Consumer Key - A unique identifier that serves as a key value for authentication purposes. It is used to identify the application or user making the API request and is essential for establishing a secure connection.
  • Consumer Secret - A confidential password used in conjunction with the Consumer Key to authenticate API requests. It ensures that the API calls are made securely and are authorized only for legitimate users or applications.

To obtain these credentials, refer to the Request Access section. These credentials are required to generate an access token, which is essential for authorizing and securing all API interactions. The process of generating the access token varies based on the user role: Partner or Customer.

Endpoint URL: GET https://api.zebra.com/v2/oauth/token/access

Partner Authentication

This section is intended for Partners only and provides authentication guidelines specific to partner use cases.

Authentication Workflow:

  1. Partner Setup:
    • The partner application must provide the apiKey (Consumer Key) and apiSecret (Consumer Secret) for authentication.
    • The customer must have a valid Zebra account registered on the Zebra Developer Portal.
  2. Customer Authentication:
    • The customer authenticates using Zebra Single Sign-On (SSO) with the registered account.
    • The customer’s registered email must be shared with the partner to activate API access.
    • Partners must notify Zebra to activate the REST API for the specified customer account before the customer can make API calls.
  3. Access Token Generation:
    • Upon successful authentication, the Authenticate API generates an access token, which is used to authorize all subsequent API requests. When the token expires, a new token must be generated.

3-Legged OAuth Flow:

The following steps demonstrate how the 3-Legged OAuth flow works to generate an access token:

  1. Fetch Device Authorization Code:

    • Make a request to retrieve the verification_uri and device_code.

    Sample Request (bash):

    curl --location --request POST 'https://uat-api.zebra.com/v1/user/device_authorization' \
--header 'Authorization: Basic <EncodedKey>'

    Sample Response (json):

    {
    "user_code": "TNDP-6JHG",
    "device_code": "uT4J6eTvmE3IeE32k4ZLrYeEbAtKnjOCKlJYThisrE",
    "verification_uri": "https://pi.zebra.com/as/user_authz.oauth2",
    "verification_uri_complete": "https://pi.zebra.com/as/user_authz.oauth2?user_code=TNDP-6JHG",
    "expires_in": 600
}
    • The verification_uri and verification_uri_complete are used for customer authentication.

  2. Generate Access Token: After the customer completes the verification process using the verification_uri_complete, use the device_code to generate an access token.

    Sample Request (bash):

    curl --location 'https://uat-api.zebra.com/v1/user/token' \
--header 'Content-Type: application/x-www-form-urlencoded' \
--header 'Authorization: Basic <EncodedKey>' \
--data-urlencode 'grant_type=urn:ietf:params:oauth:grant-type:device_code' \
--data-urlencode 'device_code=<DeviceCode>'

    Sample Response (json):

    {
    "access_token": "g1bylfjXd7Jd6mGNumJcw3ZxsWBr",
    "refresh_token": "XzrDJfu3pG51rTsW0PmqCnx2H52KxqgEKYbRdF9Shv",
    "expires_in": 3599,
    "token_type": "Bearer",
    "sub": "sample@gmail.com"
}

The access token must be included in the Authorization header of ALL API requests.

Example (bash):

    --header 'Authorization: Bearer <access_token>'

When the token expires, a new token must be generated by repeating the 3-Legged OAuth flow.

For more information on using 3-Legged OAuth, refer to the Zebra OTA API documentation (beginning on page 10).

Get Device Details

The Get Device Details API retrieves information about devices managed within the Device Guardian system. This API allows users to fetch device details such as device status, connected access point, logged-in users, and more. It supports querying all devices or filtering specific ones based on various parameters.

Capabilities:

  1. Batch Retrieval - Retrieve all devices or a filtered subset of devices from the database. Each API call can take action on up to 300 devices per request.
  2. Filtering Options - Query devices based on specific attributes such as model_serialNumber, deviceState, site, etc.
  3. Pagination Support - This API supports paginated responses to navigate through large datasets.
  4. Retrieve Device Information - Fetch detailed device information, including:
    • Device State - Current operational status of the device (e.g., IN_SERVICE, TO_BE_FOUND, etc.).
    • Connected Access Point - The friendly name of the access point to which the device is connected.
    • Last Logged in User - The most recent user who logged in to the device.
    • Current Logged in User: The user currently logged in.
    • ModelNumber - Identifier for the device model.
    • SerialNumber - Unique serial number of the device.
    • Site Name - The site to which the device belongs.

Endpoint URL: POST /v1/devices

Set Device Details

The Set Device Details API allows updating the state and assigned site of a single device or multiple devices, with support for updating up to 300 devices per request. This API also enables modifications to check-in and check-out timestamps and statuses.

A device's state can only transition to a new state if it satisfies the eligibility criteria. If a transition is not allowed, the operation will fail and return detailed error information. State transitions are contingent on the device's current state, and certain states may not be eligible for transition to others.

Note: The siteName attribute must be updated independently and should not be combined with modifications to other device attributes.

Supported Device States:

  • TO_BE_FOUND
  • INSERVICE
  • NOT_FOUND
  • FOUND
  • BEING_FOUND
