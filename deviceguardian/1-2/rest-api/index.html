REST APIs

Device Guardian + Access Management 1.2

All references to Device Guardian also apply to Device Guardian Access Management within this guide.

Overview

The Device Guardian REST API provides comprehensive features for managing resources, including sites, access points, and devices. It supports key operations including creating, updating, and deleting resources, along with features for managing device recommissioning and decommissioning processes.

Key Features:

  • Resource Management - The API supports operations for creating, updating, and deleting multiple resources in a single request.
  • Batch Processing - A single API request can process a maximum of 300 resource operations, including creation, updates, or deletions.
  • Secure Access - All API calls require an authenticated token to maintain secure communication between your application and the API. Authentication is secured through 3-Legged OAuth authentication, ensuring safe API usage.

Base URL

All API requests must be directed to the following base URL: https://api.zebra.com/partner-device-guardian

This URL serves as the foundational endpoint for all API requests related to the Device Guardian services. Ensure that API calls are constructed using this base URL to accurately access the desired resources and functionalities.

API Version

The current version of the API is v1.0.0.

  • Each API response includes this version number in the result, ensuring consistency and compatibility with your application.
  • Versioning is important for maintaining alignment with API updates and changes. Verify the API version to avoid disruptions caused by potential updates.

Definitions of Terms

  • bodyData - A JSON object that must be included in the request. It contains the data and parameters required for the API to perform the desired operation, ensuring the request is processed correctly.
  • access_token - An authentication token obtained after successfully invoking the Authenticate API. This token is crucial for authorizing and securing API calls, ensuring only authenticated users can access the API’s functionalities and maintain system integrity.

Authenticate

The Authenticate API enables secure access to the APIs by implementing the 3-Legged OAuth flow. It provides partners and customers with a trusted method to authenticate and interact with APIs while protecting sensitive credentials. Authentication through 3-Legged OAuth is mandatory for accessing any REST API, ensuring that all interactions between your application and the API are secure and protected.

Credentials Required:

  • Consumer Key - A unique identifier that serves as a key value for authentication purposes. It is used to identify the application or user making the API request and is essential for establishing a secure connection.
  • Consumer Secret - A confidential password used in conjunction with the Consumer Key to authenticate API requests. It ensures that the API calls are made securely and are authorized only for legitimate users or applications.

These credentials are essential for obtaining an access token, which is required to authorize and secure all API interactions.

Endpoint URL: GET https://api.zebra.com/v2/oauth/token/access

Authentication Workflow

  1. Partner Setup:
    • The partner application must provide the apiKey (Consumer Key) and apiSecret (Consumer Secret) for authentication.
    • The customer must have a valid Zebra account registered on the Zebra Developer Portal.
  2. Customer Authentication:
    • The customer authenticates using Zebra Single Sign-On (SSO) with the registered account.
    • The customer’s registered email must be shared with the partner to activate API access.
    • Partners must notify Zebra to activate the REST API for the specified customer account before the customer can make API calls.
  3. Access Token Generation:
    • Upon successful authentication, the Authenticate API generates an access token, which is used to authorize all subsequent API requests. When the token expires, a new token must be generated.

3-Legged OAuth Flow

The following steps demonstrate how the 3-Legged OAuth flow works to generate an access token:

  1. Fetch Device Authorization Code:

    • Make a request to retrieve the verification_uri and device_code.

    Sample Request (bash):

    curl --location --request POST 'https://uat-api.zebra.com/v1/user/device_authorization' \
--header 'Authorization: Basic <EncodedKey>'

    Sample Response (json):

    {
    "user_code": "TNDP-6JHG",
    "device_code": "uT4J6eTvmE3IeE32k4ZLrYeEbAtKnjOCKlJYThisrE",
    "verification_uri": "https://pi.zebra.com/as/user_authz.oauth2",
    "verification_uri_complete": "https://pi.zebra.com/as/user_authz.oauth2?user_code=TNDP-6JHG",
    "expires_in": 600
}
    • The verification_uri and verification_uri_complete are used for customer authentication.

  2. Generate Access Token: After the customer completes the verification process using the verification_uri_complete, use the device_code to generate an access token.

    Sample Request (bash):

    curl --location 'https://uat-api.zebra.com/v1/user/token' \
--header 'Content-Type: application/x-www-form-urlencoded' \
--header 'Authorization: Basic <EncodedKey>' \
--data-urlencode 'grant_type=urn:ietf:params:oauth:grant-type:device_code' \
--data-urlencode 'device_code=<DeviceCode>'

    Sample Response (json):

    {
    "access_token": "g1bylfjXd7Jd6mGNumJcw3ZxsWBr",
    "refresh_token": "XzrDJfu3pG51rTsW0PmqCnx2H52KxqgEKYbRdF9Shv",
    "expires_in": 3599,
    "token_type": "Bearer",
    "sub": "sample@gmail.com"
}

The access token must be included in the Authorization header of all API requests.

Example (bash):

    --header 'Authorization: Bearer <access_token>'

When the token expires, a new token must be generated by repeating the 3-Legged OAuth flow.

For more information on using 3-Legged OAuth, refer to the Zebra OTA API documentation (beginning on page 10).

Get Device Details

The Get Device Details API retrieves information about devices managed within the Device Guardian system. This API allows users to fetch device details such as device status, connected access point, logged-in users, and more. It supports querying all devices or filtering specific ones based on various parameters.

Capabilities:

  1. Batch Retrieval - Retrieve all devices or a filtered list of devices from the database. Each API call can retrieve up to 300 devices at a time.
  2. Filtering Options - Query devices based on specific attributes such as model_serialNumber, deviceState, site, etc.
  3. Pagination Support - This API supports paginated responses to navigate through large datasets.
  4. Retrieve Device Information - Fetch detailed device information, including: - Device State - Current operational status of the device (e.g., IN_SERVICE, TO_BE_FOUND, etc.). - Connected Access Point - The friendly name of the access point to which the device is connected. - Last Logged in User - The most recent user who logged in to the device. - Current Logged in User: The user currently logged in. - ModelNumber - Identifier for the device model. - SerialNumber - Unique serial number of the device. - Site Name - The site to which the device belongs. - Inuse - Indicates whether the device is currently in use.

Endpoint URL: POST /v1/devices

Set Device Details

The Set Device Details API allows updating the device state and its assigned site for a single device or a list of devices. This API also supports modifying check-in and check-out time and status. Each device's state can only transition to a new state if it meets the eligibility criteria; otherwise, the operation fails and provides relevant error details.

Supported Device States:

  • TO_BE_FOUND
  • INSERVICE
  • NOT_FOUND
  • FOUND
  • BEING_FOUND

Endpoint URL: PUT {BASE_URL}/v1/devices
