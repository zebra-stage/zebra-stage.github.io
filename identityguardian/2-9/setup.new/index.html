Setup

Identity Guardian 2.9

Overview

Identity Guardian provides secure, frictionless device access while safeguarding user data. To accommodate diverse operational workflows, the application offers two primary device deployment architectures, each with distinct authentication workflows.

  • Shared Device: For environments where multiple users share a device pool. Users can unlock a device by scanning a provisioned barcode or tapping an NFC card containing encrypted credentials, followed by optional verification methods (facial biometrics, passcode, or SSO). Alternatively, if Guardian Safe is configured, users can bypass physical credentials entirely and authenticate directly via SSO or a Zebra DNA Cloud Passcode.
  • Personally Assigned Device: For dedicated, single-user devices. After initial enrollment, access is streamlined using facial biometrics, SSO or a local passcode. Personal data is securely embedded within the Android framework, ensuring complete privacy and preventing organizational access.

To use Identity Guardian, users must enroll in the system before signing in to authenticate their devices. Integration with identity providers (IdPs) enables single sign-on (SSO), streamlining application access with a single login.

Identity Guardian can be installed and configured using one of the following methods:

  • Zebra DNA Cloud: Navigate to My Apps > Zebra Collection to install and manage the application.
  • Enterprise Mobility Management (EMM) system: Administrators can configure Identity Guardian using Managed Configurations. This enables the setup of usage policies, device controls, and customized device restrictions for secure and tailored access.

The Zebra DNA Cloud dashboard provides administrators with actionable insights, such as sign-in/sign-out events and usage durations, fostering accountability and efficient management. These insights help organization ensure operational efficiency and accountability in managing devices.

Device Usage Models

Identity Guardian offers Multi-Factor Authentication (MFA) across all deployment scenarios. Regardless of the Device Usage Model chosen, administrators can enforce customized security utilizing Single Sign-On (SSO), facial biometrics, and secure passcodes. Before deploying Identity Guardian, administrators must determine the appropriate Device Usage Model for their workforce. The configuration and deployment architecture rely on which model is selected.

For Shared Devices, there are two modes:

  • Self-Enrollment requires users to register to the system on a dedicated device before using a separate authentication device for daily operations.
  • Standard Enrollment assumes the designated enrollment device is already in a secure, supervised location.

Credential Storage Mechanisms (Shared Devices)

Organizations deploying a shared device model must select a physical credential mechanism (barcode or NFC card) to store encrypted user data, ensuring secure and personalized access. Personal data is not stored in the cloud; it remains entirely localized to the physical credential.

An exception to physical credential storage applies specifically to the use of Guardian Safe combined with an MPIN and either SSO or a Zebra DNA Cloud Passcode.

User data is encrypted using an enrollment key. This key is an encrypted public key derived from the organization’s certificate and is applied via Identity Guardian Managed Configuration. For guidance on generating this key, refer to Enrollment Configuration. It is essential to ensure that the dedicated enrollment device is provisioned with the correct Enrollment Configuration prior to generating credentials.

Barcode Storage

User data is securely encrypted into a personal barcode format. Organizations can configure these barcodes to expire after a specific timeframe to enforce credential rotation.

  • Standard Enrollment: The generated barcode is temporarily stored on the enrollment device within the /enterprise/usr/Profiles folder. It must be printed or shared via the zCreator app within 24 hours. After this period, the file is automatically purged to prevent data retention.
  • Self-Enrollment: The temporary storage mechanism is bypassed. The user must manually print or share the barcode immediately upon generating it on the device.

NFC Card Storage

User data is securely written to a compatible corporate NFC card. For daily use, personnel simply tap the NFC card to the device to authenticate and gain access. No additional authentication factors (such as facial scan or passcode entry) are required, unless configured by the admin.

  • Standard Enrollment: The encrypted NFC payload is temporarily saved on the enrollment device and remains accessible via the zCreator application. This payload must be written to an NFC card within 24 hours. After 24 hours, the data is automatically purged.
  • Self-Enrollment: The temporary storage mechanism is bypassed. The user must manually prompt the device to write the data to their NFC card immediately upon generation.

Requirements

Before beginning deployment, ensure the environment and devices meet the following requirements:

  • Supported Hardware: Compatible Zebra devices. A front-facing camera is required for facial biometrics and an NFC reader is required for NFC authentication. For compatiable devices, see Zebra Support Portal.
  • Operating System: Devices must run Android 11 or higher.
  • Management Console: Access to aan Enterprise Mobility Management (EMM) system or Zebra DNA Cloud to deploy Managed Configurations.
    • Enrollment State: Devices must be enrolled as Dedicated Device (Device Owner)
      NOTE: Corporate-Owned, Personally-Enabled (COPE) enrolled devices are currently not supported.
  • Identity Provider (IdP): An active corporate IdP (such as Microsoft Entra ID, PingID, or OKTA) utilizing OAUTH and OAUTH+OIDC authentication protocols is required if deploying Single Sign-On (SSO) authentication.
  • Licensing:
    • Mobility DNA Enterprise license: Required for basic functionality on Zebra Professional-series devices.
    • Identity Guardian license: Required for advanced features including facial biometric authentication, Single Sign-On (SSO), and device API support; see Licensing.


zCreator Requirements (Optional)

zCreator is a companion application that is required only if printing barcodes or formatting NFC authentication cards. It requires Identity Guardian version 2.2 or higher.

Supported Printers

  • Zebra Models: ZD620, ZD621
  • Canon Model: Canon PIXMA E560 Series (300 dpi)

NFC Card Specifications

  • Compatible Card Types: Mifare, DESFire EV3, Salto
  • Operating Frequency: NFC cards must operate at 13.56 MHz, the standard frequency for NFC technology.
  • Supported Format: Only NDEF format is supported. Note that the application overwrites any existing data present on the card.
  • Memory Requirements: The required data storage scales based on the number of facial captures:
    • With 1 facial capture: ~1232 bytes
    • With 2 facial captures: ~1744 bytes
    • With 3 facial captures: ~2256 bytes
  • Minimum Capacity: A minimum card capacity of 2 KB is required to successfully store user enrollment data.

Downloads & Update Management

Identity Guardian and its companion application, zCreator, are available through two primary distribution channels depending on the enterprise environment:

  • Managed Google Play Store (Recommended): The standard method for Enterprise Mobility Management (EMM) deployments. Administrators approve and distribute the applications directly through the EMM console for seamless provisioning.
  • Zebra Support Portal: For secure environments operating without Google Mobile Services (GMS) or those requiring manual APK deployment, download the installation files directly from the Zebra Support Portal.

Note: The installation order of Identity Guardian and zCreator does not affect functionality.

Google Play Store Updates

When utilizing the Managed Google Play Store for application distribution, Google Play Store updates are disabled by default starting with the April 2026 LifeGuard releases for Android 14 and Android 15. This change replaces the previous default-enabled state to prevent automatic updates from interrupting provisioning and to maintain better version control. For supported platforms and devices, see the Zebra Support Portal. While updates are disabled by default, administrators retain full control over application management.

Notes:

  • This change also restricts standard update methods, including manual installation, manual updates via the Google Play Store, and Android Debug Bridge (ADB).
  • This change is not applicable for Android 11 and Android 13, as Play Store updates are enabled by default on these versions.

Administrator Controls: Modify the default setting at any time using MX App Manager with the AllowApplicationUpgrade or DisallowApplicationUpgrade option with one of the following methods:

  • Configuration via MX App Manager: Use this method to submit the XML through the EMM platform to enable/disable automatic updates for Google Play:

    • Enable Automatic Updates: Use the AllowApplicationUpgrade option. Download the Enable_AutoUpgrade_IG.xml.
    • Disable Automatic Updates: Use the DisallowApplicationUpgrade option. Download the Disable_AutoUpgrade_IG.xml.

  • Configuration via Zebra OEMConfig: If using Zebra OEMConfig with the EMM system, follow these steps:

    1. In the EMM system, create a new setting profile using Zebra OEMConfig.

    2. Find the System Configuration section.

    3. Look for the Pass-Through Command field and copy/paste this code to enable automatic updates:

      <wap-provisioningdoc>
<characteristic version="10.3" type="AppMgr">
<parm name="Action" value="AllowApplicationUpgrade"/>
<parm name="Package" value="com.zebra.mdna.els"/>
</characteristic>
</wap-provisioningdoc>

    To disable automatic updates, use:

    <wap-provisioningdoc>
<characteristic version="10.3" type="AppMgr">
<parm name="Action" value="DisallowApplicationUpgrade"/>
<parm name="Package" value="com.zebra.mdna.els"/>
</characteristic>
</wap-provisioningdoc>

Behavior Persistence Note: This setting persists after a device reboot or an OTA upgrade.

Alternative Update Methods: Even with Google Play updates disabled, applications can still be updated directly through the following methods:

  • Zebra DNA Cloud: Centrally manage, schedule, and push Identity Guardian updates directly to the device fleet.
  • EMM Deployment: Most EMMs supporting Android Enterprise management (Device Owner or Profile Owner mode) allow administrators to directly push application updates to devices, bypassing the Google Play Store.
  • Zebra MX Deployment: The "Upgrade Action" within the Zebra MX App Manager can be used to upgrade applications via an EMM or the Zebra StageNow tool.

Prerequisites

Complete the following infrastructure and device-level configurations before deploying the Identity Guardian Managed Configurations.

1. Generate Custom Security Keys (Critical)

This section applies exclusively to Shared Devices.

In Shared Device deployments, user data — including facial capture and passcodes - is secured using cryptography. To comply with enterprise security standards and ensure data privacy, administrators must replace the default keys with a custom RSA key pair and encrypt it using the Zebra Data Encryption Tool (DET) prior to EMM deployment

Generate a unique RSA key pair using OpenSSL or a trusted corporate Certificate Authority (CA) before deploying the application.

  • Enrollment Key (Public): Deployed within the Enrollment profile. The application utilizes this key to securely encrypt the user's data into the generated barcode or NFC credential.
  • Authentication Key (Private): Deployed within the Authentication profile. The application utilizes this key to locally decrypt the user data from the credential during the login process.

Cryptographic Key Rotation (Alternate Key)

To allow organizations to update their security keys without disrupting active users, Managed Configuration includes an Alternate Authentication Key parameter. During a key rotation event, deploying the new key as the Alternate establishes a migration window where both the previous and new keys remain active. This ensures zero downtime for shift workers holding credentials generated under the previous key architecture. This feature is configured within the Authentication Configuration.

2. Configure Single Sign-On (SSO) (Optional)

This section is mandatory for Shared Device (Self-Enrollment), but optional for Shared Device (Standard Enrollment). If SSO is not required, skip this step.

Identity Guardian acts as an authentication broker for Identity Providers (IdPs), such as Microsoft Entra ID, PingID, and Okta, utilizing OAuth 2.0 and OIDC protocols. By leveraging SSO, users log in only once to gain access to multiple applications, eliminating the friction of managing separate credentials for every app.

Microsoft Authenticator and Custom Chrome Tabs are supported to act as brokers for the SSO authentication process. Refer to the Configure SSO reference section for general IdP integration guidance, and review the SSO Requirements before proceeding.

Continuous SSO Access

To eliminate repetitive login prompts during an active shift, administrators can configure the system to require full SSO authentication only once, utilizing faster localized methods (like facial biometrics or a passcode) for subsequent screen unlocks. For detailed implementation steps, refer to Continuous SSO Access in the User Guide.

3. Disable Safe Mode

Identity Guardian is bypassed if a user boots the device into Safe Mode. Deploy a Zebra StageNow profile or EMM custom XML to disable Safe Mode on all target devices before application deployment. Download the required XML and StageNow barcode based on the Android version:

4. Configure DataWedge

Applies when barcodes are used for authentication.

Identity Guardian relies on Zebra's built-in DataWedge app to scan physical barcode credentials seamlessly. A default profile named "IdentityGuardian" is automatically generated upon application initialization.

For most deployments, this default is sufficient. However, if an organization requires specific scanner behaviors, administrators must manually create and export a custom profile.

Whether customizing the default profile or building a new one, follow the steps below. When exporting the final configuration via the DataWedge database file, ensure the active "IdentityGuardian" profile is included alongside any other enterprise scanner profiles.

  1. Open DataWedge.
  2. From the top right options menu, select New Profile.
  3. Enter a profile name, such as "IdentityGuardian."
  4. Locate the new profile in the list and tap its name.
  5. The profile is enabled by default. Tap Associated apps.
  6. From the top right options menu, select New app/activity.
  7. Scroll through the app list and select com.zebra.mdna.els.
  8. Activities associated with the app is visible. From this list, select:
    • com.zebra.mdna.els.authentication.AuthenticationActivity
    • com.zebra.mdna.els.enrollment.LogoutActivity
  9. The selected activities are displayed.
  10. Tap back to return to the IdentityGuardian profile screen. Ensure Barcode Input is enabled, then tap Scanner selection.
  11. Ensure that the Auto option is selected.
  12. Tap back to return to the IdentityGuardian profile screen. Tap on Configure scanner settings.
  13. Tap on Reader params.
  14. Tap on Beam timer.
  15. Set the Beam Timer to 15000 ms and click OK.
    • Note: Zebra recommends this 15-second timeout to allow sufficient time for scanning. This value, already set in the default profile, is particularly helpful in environments with vehicle-mounted devices where more time maay be needed to align the scanner.
  16. Tap the back button to return to the IdentityGuardian profile screen. Scroll to Keystroke output and ensure it is disabled.
  17. Scroll to Intent Output and ensure it is enabled. Tap Intent action.
  18. Enter the intent action name "com.zebra.mdna.els.DATA" and tap OK.
  19. Tap Intent category. Enter "android.intent.category.DEFAULT" and tap OK.
  20. Tap Intent delivery. Select Broadcast intent.
  21. Verify all settings.
  22. Export the profile and deploy it to the devices. For instructions, see DataWedge Profile Deployment.

5. Distribute Applications

Applies to devices enrolled in Android Enterprise receiving Managed Configurations via the Managed Google Play Store.

Approve and distribute the following packages via the Managed Google Play Store to the target device groups based on the chosen deployment architecture:

  • Zebra Identity Guardian (com.zebra.mdna.els): The core authentication application.
  • zCreator (com.zebra.mdna.zcreator): Required only if printing barcodes or formatting NFC cards.
  • Microsoft Authenticator (com.microsoft.identity.client): Required only if using Entra ID SSO.

Note: When utilizing the 42Gears EMM system, apps installed via Zebra DNA Cloud in app update mode must be set as high priority.

6. Auto-Launch Intent (Critical)

Identity Guardian must be launched at least once after installation for Managed Configurations to be applied.

Managed Configurations remain inactive until the application initializes. If the EMM platform cannot automatically launch applications immediately post-installation, use one of the following methods to initialize the app:

  • Execute via ADB Shell (e.g., from another app): 

    adb shell am start -n com.zebra.mdna.els/com.zebra.mdna.els.userEnrollment.activity.EnrollmentActivity

  • Deploy via EMM Intent: 

    mode=explicit,broadcast=false,action=android.intent.action.MAIN,package=com.zebra.mdna.els,class=com.zebra.mdna.els.userEnrollment.activity.EnrollmentActivity

7. Configure zCreator (Print & Share Barcode)

Applies when barcodes are used for authentication.

The zCreator app, a companion to Identity Guardian, is utilized exclusively within Shared Device environments to provision physical user credentials.

During the enrollment phase (for both Standard and Self-Enrollment workflows), zCreator securely encrypts captured facial vectors or passcodes and generates a unique 2D barcode for the user. The application provides features such as on-screen preview, resizing, and direct printing to ensure readability, while also allowing the barcode to be securely shared via Gmail or Bluetooth.

Note: zCreator also handles NFC credential writing. If deploying NFC cards instead of barcodes, skip this printing configuration and refer to the NFC-Based Enrollment section.

Key features:

  • Streamlined Printing Options: Print barcodes directly to supported Zebra and Canon printers via WiFi, Bluetooth, and Ethernet, with:
    • Fixed barcode sizes (3 cm, 4 cm, 5 cm, 6 cm) for consistent printing.
    • Centralized admin Managed Configurations for streamlined deployment.
    • Direct printing to supported Zebra and Canon printers for faster workflows.
    • NFC Tap & Pair for instant connection with NFC-enabled Zebra printers.
  • Barcode Sharing:
    • Securely share barcodes using Gmail, Bluetooth, Canon Print, or by saving them as a PDF through the Print option.

For guidance on installation and configuration, refer to the Self-Enrollment and Standard Enrollment sections. Instructions for manual configuration post-installation are provided below.

Important Notes for Printing Operation:

  • For compatible printers, see Requirements.
  • The printer and device must be on the same network.
  • Adjust the QR Code Size (specific to Zebra printers) and printer paper size (applicable to all compatible printers) as needed to prevent the barcode from being cropped.
  • Calibrate the printer for proper printing.

Install & Configure Printer Apps:

  • Canon Printers: For printing with Canon printers, download and install the Canon Print application from the Play Store. Proper calibration and adjustment of the Canon printer settings are essential to ensure the barcode is printed correctly.
  • Zebra Printers: For printing with Zebra printers, download and install the Zebra Printer Utility application from the Play Store. Recommended Settings for QR Code printing:
    • Printing speed: Should not exceed 4.
    • Darkness: Set between 10 and 15.
    • Supported paper sizes:
      • 2-inch (5 cm) width paper: Allows QR Codes of 3 or 4 cm with accompanying text.
      • 3-inch (7.62 cm) width paper: Allows QR Codes of 3, 4, 5, and 6 cm with accompanying text.

Managed Configurations Setup

This section provides general guidance for implementing Managed Configurations through an EMM or Zebra DNA Cloud. These instructions focus on the mandatory baseline parameters required to establish core functionality.

Select the deployment path (A, B, or C) that matches the chosen Device Usage Model. Adjust any unlisted parameters based on organizational policies, and push the final configuration to the target devices.

Deployment Resources:

  • Managed Configuration Reference: The steps below serve as a high-level guide for basic deployment. For detailed descriptions of every available parameter, refer to the comprehensive Managed Configuration guide.
  • Common Configuration Workflows: Guides how to combine baseline parameters to achieve standard operational goals, such as Multi-Factor Authentication (MFA), NFC enrollment, and offline authentication.
  • Supplemental Configurations Reference: An index of advanced security capabilities and niche operational features — such as Auto Logout, Guardian Safe, and Device Alarms — with direct links to implementation instructions in the User Guide.
  • EMM Video Tutorials: Consult the EMM Guidance section to review step-by-step deployment videos specifically tailored to the EMM platform.

Path A: Shared Device (Standard Enrollment)

This model requires creating separate Managed Configuration profiles assigned to two distinct device groups: the dedicated enrollment device and the daily-use authentication devices. An additional zCreator profile must be configured for the use of barcodes or NFC cards.

Enrollment Profile Settings: Deploy this profile to the dedicated Enrollment Device group.

  • Usage Mode:
    • Application Mode: ENROLLMENT
  • Enrollment Configurations:
    • Enrollment Key: [Enter the enrollment key to encrypt user data.]
    • Number of facial images to be enrolled: [This applies only if facial authentication is required (Allow facial opt-out is set to false). For barcode printing, enter 1, as it is the only valid value for this scenario. For other use cases, enter the appropriate value based on your requirements.]
    • Allow opening barcode during enrollment: [Enable this option to allow barcode printing during the enrollment process. This displays the Print button at the last step of enrollment.]
    • Enrollment Storage Method: [Select NFC or Barcode]

    • QR Code Size: [Specify the size of the QR Code to be printed.]

  • Usage Mode:
    • Application Mode: AUTHENTICATION
  • Lock Screen Configuration:

Common Configurations

The following use cases represent common operational requirements and detail the specific Managed Configuration parameters required to achieve them.


User Enrollment and Onboarding

This section covers how employee identities and credentials are initially established within the system.

Customizing Enrollment Methods (Shared Devices)

Objective: Define exactly how users capture their credentials during the initial enrollment phase.

Implementation: Within the Enrollment Configuration, set the Storage Method to generate a barcode or prompt for an NFC write.

  • With barcode, adjust the parameters for Facial Biometrics (image counts, opt-outs), mandate a Corporate PIN/Passcode.
  • With NFC cards, bypass the facial biometric scan or passcode entry and unlock the device with the simple tap of the NFC card; for more information, see NFC Card.

Enforcing Corporate Compliance (Terms & Conditions)

Objective: Mandate that employees review and accept corporate IT usage policies or biometric data consent forms prior to accessing enterprise hardware.

Implementation: Within the Enrollment Configuration, enable the Terms and Conditions setting and input the custom corporate policy text.

Workflow: During the initial enrollment phase, the policy text displays. The user must explicitly accept the terms before the application captures biometric data or issues a credential.

NFC-Based Enrollment and Authentication

Objective: Enable touchless enrollment and daily authentication using corporate NFC cards instead of printed barcodes.

Implementation:

  • In Enrollment Configuration, set Enrollment Storage Method to NFC.
  • In Authentication Configuration set Comparison Source to NFC.

Workflow: During enrollment, the zCreator app prompts the device to write the encrypted data to the NFC card. For daily use, the user taps the NFC card to the back of the device to initiate the login process.

Zebra DNA Cloud-Based User Authentication

Objective: Authenticate users against a central database in Zebra DNA Cloud, eliminating the need for local, on-device biometric or barcode enrollment workflows.

Implementation: In Zebra DNA Cloud, follow instructions to Manage Users in zDNA. Within the Authentication Configuration, set the following, then select the remaining options as needed:

  • Comparison Source: CLOUD
  • Primary Authentication Factor: CLOUD_PASSCODE
  • Fallback Authentication Method: ADMIN BYPASS PASSCODE

Workflow: Administrators centralize user accounts within the Zebra DNA Cloud console. On the device, the user enters a cloud passcode. Identity Guardian verifies the credentials directly against the backend database in real-time.


Daily Authentication and Access Control

This section dictates how users unlock devices during standard daily operations.

Multi-Factor Authentication (MFA) and Authentication Schemes

Objective: Enforce strict access control by requiring a customized mix of Single Sign-On (SSO), Passcodes, and Facial Biometrics.

Implementation: Within the Authentication Configuration, select SSO as the Primary Authentication Factor and select FACE or PASSCODE as the Secondary Authentication Factor.

Workflow: A user initiates login via SSO. Upon success, the system immediately requests the secondary factor. The device only unlocks when both factors are verified.

Managing Fast Access After Initial SSO Login

Objective: Streamline device unlocks during a shift without requiring full SSO authentication for every screen wake event.

Implementation: In SSO Authentication Configuration, set Temp ID to true.

Workflow: The user authenticates once using full SSO credentials at the start of a shift. For subsequent device unlocks during that same shift, the system bypasses the full SSO prompt and utilizes faster configured methods (like PIN or facial biometrics).

Operating in Disconnected Environments (Offline Authentication)

Objective: Ensure shift workers maintain device access when operating in facilities with dead zones or secure isolated networks.

Implementation: Utilize the Shared Device Model with the Enrollment Storage Method set to BARCODE or NFC. Ensure SSO is not mandated for the daily unlock sequence.

Workflow: Because encrypted biometric data is stored directly on the physical barcode or NFC card, the device processes the decryption and facial match entirely locally, requiring no backend network connection.


Shared Device and Shift Management

This section covers data privacy and session lifecycle management between different workers.

Enforcing Credential Rotation (Barcode Expiration)

Objective: Mitigate the security risk of lost or copied physical credentials by preventing the long-term use of printed barcodes.

Implementation: Within the Enrollment Configuration, set the Barcode Expiration parameter to a specific hourly limit (e.g., 12 or 24 hours).

Workflow: A generated barcode remains valid only for the configured duration. Once expired, subsequent authentication attempts fail, forcing the employee to generate a new credential.


Integrations and UI Customization

This section covers how Identity Guardian interacts with the broader software ecosystem.

Role-Based Access and Integrations (Enterprise Home Screen)

Objective: Customize the device experience and application availability based on an employee’s organizational role.

Implementation: Deploy Identity Guardian alongside Zebra Enterprise Home Screen (EHS). Ensure the Role Data retrieved during enrollment or SSO mapping matches the role definitions within EHS.

Workflow: Upon authentication, EHS dynamically reads the assigned role and displays a customized application launcher tailored specifically to that user’s access level.

      • Role Identifier - Establishes links between roles in SSO responses and their corresponding roles within the Identity Guardian app.
      • Identity Guardian Role Name - Enter the Identity Guardian user role to be assigned based on SSO response during user sign-in
      • Key-value Pair for Role Assignment - Add one or more SSO key-value pairs to identify and map users to a predefined Identity Guardian user role. Click Add SSO Key-Value Pair as needed.
      • SSO Key-Value Pair - Choose whether the SSO response, which contains the user key and values, should be mapped to a corresponding user role in Identity Guardian
      • SSO Key - Enter the SSO key to map it to an Identity Guardian role.
      • SSO Value - Enter the SSO value(s) to map to the Identity Guardian role. Use commas to separate multiple entries. zDNA Cloud Managed Config

  7. See Managed Configurations to configure any other non-SSO settings.

  8. Deploy the Managed Configurations to the devices through your EMM.

When a device gets updated with the new configurations, Zebra Identity Guardian activates the lock screen. The user is then required to authenticate themselves using Single Sign-On (SSO) to gain access to the device. See User Guide for more information.


EMM Guidance

Deploy and configure Identity Guardian on devices using an Enterprise Mobility Management (EMM) system, such as Zebra DNA Cloud or a third party system, which supports Managed Configurations.

After installation, Identity Guardian must be launched for the configurations to take effect.

Generic Setup Process Using EMM:

  1. Enroll the devices to the EMM.

  2. Upload the required APKs to the EMM: This can be done via Android Enterprise or the EMM's App Store.

    • Add the Identity Guardian APK to the EMM. Ensure the Identity Guardian app is configured to auto-launch after installation. The EMM system may offer the option to auto-launch the app. If this option is not available, the following commands may be used to launch Identity Guardian:

      • For Android, such as from another app, use this command:

        adb shell am start -n com.zebra.mdna.els/com.zebra.mdna.els.userEnrollment.activity.EnrollmentActivity

      • For an EMM, such as VMware Workspace ONE UEM (AirWatch), use this command to launch via intent:

        mode=explicit,broadcast=false,action=android.intent.action.MAIN,package=com.zebra.mdna.els,class=com.zebra.mdna.els.userEnrollment.activity.EnrollmentActivity

    • If barcode printing or sharing is needed on Shared Devices, add the zCreator APK to the EMM.

      • If the EMM kiosk configuration is being used, make sure to include the zCreator app package name to the allowlist: com.zebra.mdna.zcreator

  3. Create and configure application policies: Use Managed Configurations to configure the necessary settings for Identity Guardian and zCreator (if applicable). Refer to the following guides:

    • For Shared Devices: Choose the appropriate enrollment type:
      • Standard Enrollment - Separate Enrollment and Authentication Profiles are required.
      • Self-Enrollment - Single profile combines enrollment and authentication configurations

    • For Personally Assigned Devices: Single profile combines enrollment and authentication configurations; see Personally Assigned Device.

  4. Assign and apply the application policies:

    • Shared Devices:
      • For Standard Enrollment, deploy the Enrollment Profile on the dedicated enrollment device and the Authentication Profile on the shared device.
      • For Self-Enrollment, deploy the unified Authentication Profile directly to the shared device.

    • Personally Assigned Devices: Deploy the single profile to the device.

  5. Guide Users through Enrollment and Authentication: Users proceed through the enrollment process on the device by following the prompts. After enrollment completion, they can authenticate to sign in to the device based on the Authentication Configuration settings set by their administrator.

    • Shared Devices:

    • Standard Enrollment: Users enroll on a dedicated enrollment device and authenticate on a separate shared device; see standard enrollment and authentication.

  6. Self-Enrollment: Users complete both enrollment and authentication on the same shared device; see self-enrollment and authentication.

  7. Personally Assigned Devices: Users complete both enrollment and authentication on their personally assigned device; see personal device user enrollment and authentication.

Shared Device

Authentication Process

User authentication on shared devices requires an Authentication Key, an encrypted private key derived from the organization’s certificate and configured through Identity Guardian's Managed Configuration. To support seamless cryptographic key rotation, an optional Alternate Authentication Key can also be configured. This provides a migration period where both the primary and alternate keys are active, ensuring zero downtime for end-users. For details on key generation, see Authentication Configuration. The target device must be provisioned with the proper Authentication configuration.

When a user scans their enrollment barcode or NFC card, the system decrypts the stored data using either the Authentication Key or Alternate Authentication Key.

  • Security Warning: The data cannot be decrypted or reverse-engineered on any device without the corresponding authentication key. It is essential that organizations maintain the security and confidentiality of the authentication key to ensure protection of user data.

IMPORTANT: Administrators MUST provision their own enrollment and authentication keys on the respective devices used for enrollment and authentication. Failure to do so will result in Identity Guardian using default keys, which may allow any device with those keys to decrypt the barcode, compromising security.
